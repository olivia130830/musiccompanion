# MusicCompanion V0 原型项目文档

## 项目背景

MusicCompanion 是一个"共同听音乐"的网页原型。

在听音乐时，往往没有人能陪伴和分享。本项目通过 AI 伙伴在关键时刻进行简短评论，让用户感受"有人一起听"的陪伴感。

与传统 AI 音乐工具不同，MusicCompanion 的特点是：
- **及时性**：在音乐播放中即时出现评论，而非等待用户提问
- **克制性**：只在少量关键时刻说话，不会持续干扰音乐
- **伙伴感**：像朋友一起听歌时的自然评论，不像教师讲解

## V0 版本目标

V0 是一个完全本地化的原型，验证以下假设：

1. 时间点评论能否产生"有人一起听"的感觉
2. 简短评论会不会打扰音乐体验
3. 用户看到评论后是否有回复欲望
4. 什么样的评论频率和长度比较自然
5. 即时欣赏是否与听完后的总结形成明显区别

## V0 包含的功能

### 音频播放
- ✅ 选择本地 MP3、WAV、M4A 文件
- ✅ 播放、暂停、进度条拖动
- ✅ 显示当前时间和总时长
- ✅ 音频只在浏览器本地播放，不上传服务器

### 伙伴评论
- ✅ 预设时间点出现评论（使用人工数据，非 AI 生成）
- ✅ 仅在音乐正在播放时触发
- ✅ 同一评论在单次播放中只出现一次
- ✅ 每条评论配备"我也觉得"和"我感觉不一样"反馈按钮

### 用户互动
- ✅ 用户可输入即时感受
- ✅ Enter 发送，Shift+Enter 换行
- ✅ 交互内容保存在浏览器状态中
- ✅ 空白内容无法发送

### 共同聆听记录
- ✅ AI 评论和用户回复统一记录
- ✅ 视觉上明显区分（AI 左侧，用户右侧）
- ✅ 显示发送时的音乐进度

### 换歌处理
- ✅ 切换歌曲后清空旧歌曲的所有评论和反馈
- ✅ 新歌曲可以从头开始接收评论

## 评论触发规则

### 何时触发

1. 音乐正在播放（`audio.playing === true`）
2. 当前时间达到评论预设的 `timeSeconds`（±0.5 秒误差）
3. 该评论在当前播放会话中尚未被触发过

### 何时不触发

- 音乐暂停时
- 快进跳过评论时间点后，不补发已错过的评论
- 倒退到已触发的时间点时，不重复触发
- 音频时长短于评论预设时间时，不报错

### 防重复机制

- 使用 `Set<string>` 记录已触发的评论 ID
- 不依赖 React 状态更新判断（避免闭包问题）
- React 重复渲染时不会造成重复触发

### 拖动行为

- 用户拖动进度条后，仅等待当前时间之后的评论
- 一次 `timeupdate` 事件最多触发一条评论（防批量）
- 快进和倒退都遵循相同规则

### 换歌重置

- 选择新文件后，清空 `triggeredCommentIds` Set
- 清空所有用户回复和反馈
- 清空共同聆听记录

## 安全和隐私规则

- ✅ **本地优先**: 音频文件完全在浏览器中处理，从不上传服务器
- ✅ **URL 管理**: 使用 `URL.createObjectURL()` 和 `URL.revokeObjectURL()` 防止内存泄漏
- ✅ **无存储**: 页面刷新后数据完全消失（这是正常行为，不是 BUG）
- ✅ **无追踪**: 不收集用户数据，不使用分析工具
- ✅ **开源透明**: 代码可审计，无隐藏行为

## 开发笔记

### 关键决策

1. **为什么不用组件库？**
   - V0 需要快速迭代验证
   - 深色音乐界面需要定制化设计
   - 依赖最小化

2. **为什么用普通 CSS？**
   - Tailwind 学习成本对初学者友好，但本项目已明确需求
   - 普通 CSS 更易理解修改
   - 无必要第三方依赖

3. **为什么只支持本地文件？**
   - V0 关注评论体验，不关注音乐源
   - 本地文件完全由用户控制
   - 隐私优先

4. **为什么评论是预设的？**
   - V0 验证的是交互体验，不是 AI 能力
   - 预设数据稳定可测试
   - 后续可轻松替换为真实 AI 接口

## 故障排除

### 问题: 页面加载后没有按钮
**原因**: JavaScript 未加载或出错
**解决**: 检查浏览器控制台是否有错误

### 问题: 音频无法播放
**原因**: 浏览器不支持该格式，或者文件损坏
**解决**: 尝试其他音频格式，检查文件是否完整

### 问题: 评论总是重复出现
**原因**: `useCommentScheduler` 中已触发集合未正确管理
**解决**: 检查 Set 的初始化和更新逻辑

### 问题: 内存持续增长
**原因**: Object URL 未正确释放
**解决**: 确保 `URL.revokeObjectURL()` 在组件卸载和换歌时被调用
